46 /*
47  There is just one server thread and it is the only one
48  messing with the recv buffers, so there is no need for
49  mutex on recv related data.
50 
51  Multiple threads (including the server) may send hence
52  we need to be careful about send-related data.
53 
54  When MPI is initialized we need to use init_thread with
55  multiple required.
56 
57  This RMI service operates only in (a clone of) COMM_WORLD. It easy enough
58  to extend to other communicators but the point is to have
59  only one server thread for all possible uses. You just
60  have to translate rank_in_comm into rank_in_world by
61  getting the groups from both communicators using
62  MPI_Comm_group and then creating a map from ranks in
63  comm to ranks in world using MPI_Group_translate_ranks.
64 
65  The class is a singleton ... i.e., there is only one instance of it
66  that is made the first time that you call RMI::instance().
67 
68  Handler routines should have this type
69 
70  typedef void (*rmi_handlerT)(void* buf, size_t nbyte);
71 
72  There are few user accessible routines.
73 
74  RMI::Request RMI::isend(const void* buf, size_t nbyte, int dest,
75  rmi_handlerT func, unsigned int attr=0)
76  - to send an asynchronous message
77  - RMI::Request has the same interface as SafeMPI::Request
78  (right now it is a SafeMPI::Request but this is not guaranteed)
79 
80  void RMI::begin()
81  - to start the server thread
82 
83  void RMI::end()
84  - to terminate the server thread
85 
86  bool RMI::get_debug()
87  - to get the debug flag
88 
89  void RMI::set_debug(bool)
90  - to set the debug flag
91 
92 */
